# React Application Security Vulnerabilities — Complete Guide

> Interview-ready breakdown: What each tool/technique is, how it solves the problem, how to apply it in real-time, and what output it produces.

---

## 1. Dependency Auditing

### What is it?
Your `node_modules` folder contains hundreds of third-party packages. Each of those packages can have known security flaws (called CVEs — Common Vulnerabilities and Exposures). Dependency auditing is the process of scanning all your installed packages against a database of known vulnerabilities.

### How does it solve the problem?
It maps every package in your `package-lock.json` or `yarn.lock` to the **npm advisory database** and flags any that have reported security issues — telling you the severity (low / moderate / high / critical), which package is affected, and what the fix is.

### How to apply it in real-time

**Step 1 — Run a scan:**
```bash
npm audit
```

**Step 2 — Auto-fix safe patches:**
```bash
npm audit fix
```

**Step 3 — Force-fix (use carefully, may break things):**
```bash
npm audit fix --force
```

**Step 4 — Integrate into CI/CD (GitHub Actions example):**
```yaml
# .github/workflows/security.yml
name: Security Audit

on: [push, pull_request]

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      - run: npm audit --audit-level=high
        # Fails the build if any HIGH or CRITICAL vulnerabilities are found
```

**With Snyk (deeper analysis):**
```bash
npx snyk test              # one-time scan
npx snyk monitor           # continuously monitor and alert
```

### What output it produces
```
found 3 vulnerabilities (1 moderate, 2 high)

  High: Prototype Pollution in lodash
  Package: lodash
  Dependency of: react-scripts
  Path: react-scripts > lodash
  More info: https://npmjs.com/advisories/1523
  Fix: upgrade to lodash@4.17.21
```

## 2. Static Code Analysis (ESLint + Security Plugins)

### What is it?
Static analysis means scanning your **source code without running it**. Tools read your code like a reviewer and flag patterns that are known to be dangerous — like using `eval()`, unsanitized HTML injection, or insecure regex.

### How does it solve the problem?
It catches vulnerabilities **at the time you write code**, before anything is committed or deployed. It's like having a security expert looking over your shoulder in the editor.

### How to apply it in real-time

**Step 1 — Install:**
```bash
npm install --save-dev eslint eslint-plugin-security eslint-plugin-react
```

**Step 2 — Configure `.eslintrc.js`:**
```js
module.exports = {
  plugins: ['security', 'react'],
  extends: [
    'plugin:react/recommended',
    'plugin:security/recommended'
  ],
  rules: {
    'react/no-danger': 'error',              // flags dangerouslySetInnerHTML
    'security/detect-eval-with-expression': 'error',
    'security/detect-non-literal-regexp':   'warn',
    'security/detect-object-injection':     'warn'
  }
};
```

**Step 3 — Run manually:**
```bash
npx eslint src/
```

**Step 4 — Add to CI/CD:**
```yaml
- name: Lint for security
  run: npx eslint src/ --max-warnings=0
```

**Step 5 — Real-time in VS Code:**
Install the ESLint VS Code extension → violations underline in red as you type.

### What output it produces
```
src/components/UserCard.jsx
  14:5  error  Dangerously setting inner HTML  react/no-danger
  27:3  warn   Object bracket notation with user input  security/detect-object-injection

✖ 2 problems (1 error, 1 warning)
```

## 3. SonarQube / SonarCloud (Enterprise-Grade Static Analysis)

### What is it?
SonarQube is a **dedicated code quality and security platform**. It goes deeper than ESLint — it understands data flow, tracks how user input moves through your application (called taint analysis), and detects complex patterns like SQL injection chains even in JavaScript.

### How does it solve the problem?
Where ESLint flags individual lines, SonarQube traces the **entire path of data** — e.g., it can detect that a URL query parameter flows into `innerHTML` three components away. It also tracks security hotspots, code smells, and technical debt over time.

### How to apply it in real-time

**Option A — SonarCloud (free for public repos):**
1. Go to [sonarcloud.io](https://sonarcloud.io) → connect GitHub repo
2. Add to CI/CD:

```yaml
# .github/workflows/sonar.yml
- name: SonarCloud Scan
  uses: SonarSource/sonarcloud-github-action@master
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

3. Add `sonar-project.properties` to root:
```properties
sonar.projectKey=your-project-key
sonar.organization=your-org
sonar.sources=src
sonar.javascript.lcov.reportPaths=coverage/lcov.info
```

**Option B — SonarQube (self-hosted):**
```bash
docker run -d --name sonarqube -p 9000:9000 sonarqube:community
# Then run sonar-scanner against your repo
```

### What output it produces
- Dashboard with: Bugs, Vulnerabilities, Security Hotspots, Code Smells, Coverage %
- Per-file drill-down showing exact lines with issue descriptions
- PR decoration — comments directly on GitHub PRs flagging new issues
- Quality Gate — blocks merge if thresholds are breached

## 4. Manual Code Review — What to Look For

### What is it?
No tool catches everything. Manual code review means a developer deliberately reads code with a **security mindset**, looking for patterns that tools often miss — logic flaws, auth gaps, business rule violations.

### How does it solve the problem?
Automated tools catch known patterns. Humans catch **intent mismatches** — e.g., an API endpoint that checks authentication but skips authorization, or a component that shows data to the wrong user in a multi-tenant app.

### How to apply it in real-time

**Checklist to go through during every PR review:**

```
XSS (Cross-Site Scripting)
  [ ] Is dangerouslySetInnerHTML used?
      → If yes, is the value sanitized with DOMPurify before use?
  [ ] Is user input rendered as HTML anywhere?
  [ ] Are href/src attributes ever set from user-controlled data?
      → Can be exploited: <a href="javascript:alert(1)">

Auth & Access Control
  [ ] Are protected routes wrapped with an auth guard?
  [ ] Does the frontend trust user-supplied roles without server verification?
  [ ] Are JWT tokens stored in localStorage? (bad — XSS can steal them)
      → Should be in HttpOnly cookies

Secrets & Data Leakage
  [ ] Are API keys hardcoded in any component?
  [ ] Are secrets in .env committed to git?
      → Check: cat .gitignore | grep .env
  [ ] Does any component log sensitive data to console?

Third-Party Dependencies
  [ ] Was a new package added? Is it well-maintained and widely used?
  [ ] Does it have recent CVEs? (check npmjs.com/advisories)
```

**Dangerous React pattern example:**
```jsx
// ❌ NEVER do this
function Comment({ userInput }) {
  return <div dangerouslySetInnerHTML={{ __html: userInput }} />;
}

// ✅ Sanitize first with DOMPurify
import DOMPurify from 'dompurify';

function Comment({ userInput }) {
  return <div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userInput) }} />;
}
```

### What output it produces
PR comments, rejected merges, and a shared team understanding of security patterns — no automated report, but highest signal-to-noise ratio.

## 5. Runtime / Dynamic Analysis (OWASP ZAP)

### What is it?
Dynamic Application Security Testing (DAST) means attacking your **running application** like a hacker would. OWASP ZAP (Zed Attack Proxy) is a free, open-source tool that crawls your app and fires thousands of attack payloads at it to see what sticks.

### How does it solve the problem?
Static analysis can't catch everything — some vulnerabilities only appear when the app is running (auth bypass, session fixation, open redirects, server misconfigs). ZAP finds these by actually trying to exploit them.

### How to apply it in real-time

**Basic scan (GUI):**
1. Download OWASP ZAP from [zaproxy.org](https://zaproxy.org)
2. Enter your local URL (`http://localhost:3000`)
3. Click "Automated Scan" → ZAP crawls and attacks your app

**Automated in CI/CD:**
```yaml
# .github/workflows/dast.yml
- name: ZAP Scan
  uses: zaproxy/action-full-scan@v0.8.0
  with:
    target: 'http://localhost:3000'
    rules_file_name: '.zap/rules.tsv'
    cmd_options: '-a'
```

**Using Burp Suite (manual, more powerful):**
1. Configure browser proxy → `127.0.0.1:8080`
2. Browse your app normally — Burp intercepts every request
3. Use Repeater to modify requests and test auth/token validation
4. Use Scanner to auto-attack endpoints

### What output it produces
```
WARN-NEW: X-Frame-Options Header Not Set [10020] x 1
  http://localhost:3000/

HIGH: Cross Site Scripting (Reflected) [40012] x 2
  http://localhost:3000/search?q=<script>alert(1)</script>

MEDIUM: Absence of Anti-CSRF Tokens [20012] x 4
  http://localhost:3000/api/update-profile
```

## 6. HTTP Security Headers Check

### What is it?
Security headers are HTTP response headers your server sends that instruct the browser on how to behave securely. They're often missing in React apps because developers focus on code, not server configuration.

### How does it solve the problem?
Each header prevents a class of attacks:

| Header | Attack it prevents |
|---|---|
| `Content-Security-Policy` | XSS, data injection |
| `X-Frame-Options` | Clickjacking |
| `Strict-Transport-Security` | SSL stripping, MITM |
| `X-Content-Type-Options` | MIME-type sniffing attacks |
| `Referrer-Policy` | Data leakage via Referer header |

### How to apply it in real-time

**Check your app with:**
```bash
curl -I https://yourapp.com
# or visit https://securityheaders.com
```

**Add headers in Express (API server):**
```js
const helmet = require('helmet');
app.use(helmet()); // Sets all major headers automatically
```

**Add headers in Next.js (`next.config.js`):**
```js
const securityHeaders = [
  { key: 'X-Frame-Options', value: 'DENY' },
  { key: 'X-Content-Type-Options', value: 'nosniff' },
  {
    key: 'Content-Security-Policy',
    value: "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
  },
  { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains' }
];

module.exports = {
  async headers() {
    return [{ source: '/(.*)', headers: securityHeaders }];
  }
};
```

**Add headers in Nginx:**
```nginx
add_header X-Frame-Options "DENY";
add_header X-Content-Type-Options "nosniff";
add_header Content-Security-Policy "default-src 'self'";
add_header Strict-Transport-Security "max-age=63072000";
```

### What output it produces
```
securityheaders.com Grade: A

✅ Content-Security-Policy
✅ X-Frame-Options
✅ X-Content-Type-Options
✅ Strict-Transport-Security
❌ Permissions-Policy  ← missing
❌ Referrer-Policy     ← missing
```

## 7. CI/CD Security Pipeline (Putting It All Together)

### What is it?
Integrating all the above tools into your GitHub Actions / GitLab CI pipeline so security checks run automatically on every push or pull request — without anyone having to remember to run them manually.

### How to apply it in real-time

```yaml
# .github/workflows/security-pipeline.yml
name: Security Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  dependency-audit:
    name: Dependency Audit
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
      - run: npm ci
      - run: npm audit --audit-level=high

  static-analysis:
    name: ESLint Security Scan
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      - run: npx eslint src/ --max-warnings=0

  sonarcloud:
    name: SonarCloud Analysis
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: SonarSource/sonarcloud-github-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

  snyk:
    name: Snyk Vulnerability Scan
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: snyk/actions/node@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
```

### What output it produces
- ✅ Green checkmarks on PRs when all checks pass
- ❌ Blocked merges when vulnerabilities are found
- 📧 Email/Slack alerts for new CVEs in dependencies
- PR comments from SonarCloud pointing to exact lines with issues

## Summary: The Full Security Layer Stack

```
Layer 1 — Write Time      ESLint + security plugins (catches as you type)
Layer 2 — Commit Time     Pre-commit hooks (Husky) → run ESLint + audit
Layer 3 — PR Time         CI pipeline → npm audit + SonarCloud + Snyk
Layer 4 — Deploy Time     OWASP ZAP against staging environment
Layer 5 — Runtime         Security headers + CSP on the live server
Layer 6 — Ongoing         Snyk monitor + Dependabot for new CVEs
```
