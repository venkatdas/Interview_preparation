# API Security — Complete Guide

> APIs are the backbone of every modern app. They are also the #1 attack target.
> If your API is not secured, your frontend security means nothing — attackers skip the UI entirely and call your API directly.

---

## The Core Problem

```
Your React App  ──→  /api/getUser?id=1   ──→  Returns user data
                                         
Attacker        ──→  /api/getUser?id=2   ──→  Also returns user data ← PROBLEM
                ──→  /api/getUser?id=999 ──→  Returns admin data     ← BIGGER PROBLEM

Attacker doesn't need your frontend at all.
They use Postman, curl, or a script to hit your API directly.
```

That's why every single security check must happen on the API — not the UI.

---

## 1. Authentication — Prove Who You Are

### What is it?
Every API request must carry proof of identity. The API verifies that proof before doing anything.

### How it works — JWT Flow

```
User logs in
     │
     ▼
Server verifies credentials
     │
     ▼
Server creates JWT (signed token) ──→ sent to client
     │
     ▼
Client sends JWT in every request header:
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
     │
     ▼
Server verifies signature on every request
     │
     ├── Valid   ──→ continue
     └── Invalid ──→ 401 Unauthorized
```

### Implementation

```js
const jwt = require('jsonwebtoken');
const bcrypt = require('bcrypt');

// ── LOGIN ──────────────────────────────────────────────
app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;

  // 1. Find user
  const result = await pool.query(
    'SELECT * FROM users WHERE email = $1', [email]
  );
  const user = result.rows[0];

  // 2. User not found — same message as wrong password
  //    Never reveal which one is wrong (user enumeration attack)
  if (!user) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  // 3. Compare password (bcrypt is timing-safe)
  const valid = await bcrypt.compare(password, user.password_hash);
  if (!valid) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  // 4. Issue short-lived access token (15 minutes)
  const accessToken = jwt.sign(
    { userId: user.id, role: user.role, email: user.email },
    process.env.JWT_SECRET,
    { expiresIn: '15m', issuer: 'myapp.com' }
  );

  // 5. Issue long-lived refresh token (7 days)
  const refreshToken = jwt.sign(
    { userId: user.id },
    process.env.JWT_REFRESH_SECRET,
    { expiresIn: '7d' }
  );

  // 6. Store refresh token hash in DB (so we can revoke it on logout)
  await pool.query(
    'INSERT INTO refresh_tokens (user_id, token_hash, expires_at) VALUES ($1, $2, NOW() + INTERVAL \'7 days\')',
    [user.id, await bcrypt.hash(refreshToken, 10)]
  );

  // 7. Refresh token → HttpOnly cookie (JS can't steal it)
  res.cookie('refreshToken', refreshToken, {
    httpOnly: true,
    secure: true,
    sameSite: 'strict',
    maxAge: 7 * 24 * 60 * 60 * 1000
  });

  // 8. Access token → response body (stored in memory on client)
  res.json({ accessToken });
});


// ── AUTH MIDDLEWARE ────────────────────────────────────
function authenticate(req, res, next) {
  const authHeader = req.headers['authorization'];

  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const token = authHeader.split(' ')[1];

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET, {
      issuer: 'myapp.com'  // reject tokens from other issuers
    });
    req.user = decoded;
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Token expired', code: 'TOKEN_EXPIRED' });
    }
    return res.status(403).json({ error: 'Invalid token' });
  }
}


// ── REFRESH TOKEN ──────────────────────────────────────
app.post('/api/auth/refresh', async (req, res) => {
  const refreshToken = req.cookies.refreshToken;
  if (!refreshToken) return res.status(401).json({ error: 'No refresh token' });

  try {
    // 1. Verify the refresh token
    const decoded = jwt.verify(refreshToken, process.env.JWT_REFRESH_SECRET);

    // 2. Check if token exists in DB (not revoked)
    const stored = await pool.query(
      'SELECT * FROM refresh_tokens WHERE user_id = $1 AND expires_at > NOW()',
      [decoded.userId]
    );
    if (!stored.rows[0]) {
      return res.status(403).json({ error: 'Refresh token revoked' });
    }

    // 3. Issue new access token
    const newAccessToken = jwt.sign(
      { userId: decoded.userId },
      process.env.JWT_SECRET,
      { expiresIn: '15m' }
    );

    res.json({ accessToken: newAccessToken });
  } catch {
    res.status(403).json({ error: 'Invalid refresh token' });
  }
});


// ── LOGOUT ─────────────────────────────────────────────
app.post('/api/auth/logout', authenticate, async (req, res) => {
  // Remove refresh token from DB → can't be used again
  await pool.query(
    'DELETE FROM refresh_tokens WHERE user_id = $1',
    [req.user.userId]
  );

  // Clear the cookie
  res.clearCookie('refreshToken');
  res.json({ message: 'Logged out' });
});
```

## 2. Authorization — What Are You Allowed to Do

### What is it?
Even after you verify identity (auth), you must check permissions (authz).  
This is the most commonly skipped layer.

### Three levels to check

```
Level 1 — Are you logged in?           (authentication)
Level 2 — Does your role allow this?   (role-based)
Level 3 — Does this resource belong to you? (ownership)
```

### Implementation

```js
// ── ROLE-BASED ACCESS CONTROL ──────────────────────────
function authorize(...allowedRoles) {
  return (req, res, next) => {
    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    next();
  };
}

// Public route — no auth needed
app.get('/api/products', getProductsHandler);

// Any logged in user
app.get('/api/profile', authenticate, getProfileHandler);

// Only editors and admins
app.post('/api/posts', authenticate, authorize('editor', 'admin'), createPostHandler);

// Only admins
app.delete('/api/users/:id', authenticate, authorize('admin'), deleteUserHandler);


// ── OWNERSHIP CHECK (prevent IDOR) ────────────────────
// IDOR = Insecure Direct Object Reference
// User A changing the ID in the URL to access User B's data

app.get('/api/orders/:id', authenticate, async (req, res) => {
  const order = await pool.query(
    'SELECT * FROM orders WHERE id = $1', [req.params.id]
  );

  if (!order.rows[0]) {
    return res.status(404).json({ error: 'Not found' });
  }

  // Does this order belong to the requesting user?
  if (order.rows[0].user_id !== req.user.userId && req.user.role !== 'admin') {
    // Return 404 not 403 — don't confirm the resource exists
    return res.status(404).json({ error: 'Not found' });
  }

  res.json(order.rows[0]);
});


// ── FIELD-LEVEL AUTHORIZATION ─────────────────────────
// Regular users can update name/bio. Only admins can update role.

app.put('/api/users/:id', authenticate, async (req, res) => {
  const allowedFields = ['name', 'bio', 'avatar_url'];

  // Admin gets extra fields
  if (req.user.role === 'admin') {
    allowedFields.push('role', 'is_active');
  }

  // Only pick allowed fields from request body
  const updates = {};
  allowedFields.forEach(field => {
    if (req.body[field] !== undefined) updates[field] = req.body[field];
  });

  if (Object.keys(updates).length === 0) {
    return res.status(400).json({ error: 'No valid fields to update' });
  }

  // Build safe query
  const setClause = Object.keys(updates)
    .map((key, i) => `${key} = $${i + 1}`)
    .join(', ');

  await pool.query(
    `UPDATE users SET ${setClause} WHERE id = $${Object.keys(updates).length + 1}`,
    [...Object.values(updates), req.params.id]
  );

  res.json({ message: 'Updated' });
});
```

## 3. Input Validation — Trust Nothing from the Client

### What is it?
Every value coming in — body, params, query strings, headers — must be validated before touching your database or business logic.

### Implementation

```js
const { body, param, query, validationResult } = require('express-validator');

// ── VALIDATION MIDDLEWARE FACTORY ─────────────────────
function validate(rules) {
  return [
    ...rules,
    (req, res, next) => {
      const errors = validationResult(req);
      if (!errors.isEmpty()) {
        return res.status(400).json({
          error: 'Validation failed',
          details: errors.array().map(e => ({
            field: e.path,
            message: e.msg
          }))
        });
      }
      next();
    }
  ];
}

// ── USAGE ──────────────────────────────────────────────
app.post('/api/users',
  authenticate,
  authorize('admin'),
  validate([
    body('name')
      .trim()
      .notEmpty().withMessage('Name is required')
      .isLength({ min: 2, max: 100 }).withMessage('Name: 2–100 chars')
      .matches(/^[a-zA-Z\s'-]+$/).withMessage('Name: letters only'),

    body('email')
      .isEmail().withMessage('Valid email required')
      .normalizeEmail(),

    body('password')
      .isLength({ min: 8 }).withMessage('Min 8 characters')
      .matches(/[A-Z]/).withMessage('Must contain uppercase')
      .matches(/[0-9]/).withMessage('Must contain number')
      .matches(/[^A-Za-z0-9]/).withMessage('Must contain special character'),

    body('role')
      .isIn(['user', 'editor']).withMessage('Invalid role'),
      // 'admin' is not in the list — can't create admins via this endpoint

    param('id')
      .isInt({ min: 1 }).withMessage('Invalid ID'),

    query('page')
      .optional()
      .isInt({ min: 1, max: 1000 }).withMessage('Invalid page number'),
  ]),
  createUserHandler
);
```

## 4. Rate Limiting — Stop Abuse

### What is it?
Limit how many requests an IP can make in a time window. Stops brute force, DDoS, scraping.

### Implementation

```js
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');
const redis = require('redis');

const redisClient = redis.createClient({ url: process.env.REDIS_URL });

// ── DIFFERENT LIMITS FOR DIFFERENT ROUTES ─────────────

// General API — 100 requests per 15 minutes
const generalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  standardHeaders: true,
  store: new RedisStore({ client: redisClient }),
  handler: (req, res) => {
    res.status(429).json({
      error: 'Too many requests',
      retryAfter: Math.ceil(req.rateLimit.resetTime / 1000)
    });
  }
});

// Auth routes — 5 attempts per 15 minutes
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  skipSuccessfulRequests: true,   // only counts failed attempts
  store: new RedisStore({ client: redisClient }),
  handler: (req, res) => {
    res.status(429).json({ error: 'Too many failed attempts. Try again in 15 minutes.' });
  }
});

// Sensitive actions — password reset, MFA
const sensitiveActionsLimiter = rateLimit({
  windowMs: 60 * 60 * 1000,  // 1 hour
  max: 3,
  store: new RedisStore({ client: redisClient }),
});

// Apply
app.use('/api/', generalLimiter);
app.use('/api/auth/login', authLimiter);
app.use('/api/auth/register', authLimiter);
app.use('/api/auth/forgot-password', sensitiveActionsLimiter);
app.use('/api/auth/reset-password', sensitiveActionsLimiter);
```

## 5. CORS — Control Who Can Call Your API

### What is it?
CORS controls which domains are allowed to call your API from a browser.  
Without it, any website could make requests to your API using a logged-in user's cookies.

### Implementation

```js
const cors = require('cors');

const allowedOrigins = {
  production: ['https://myapp.com', 'https://www.myapp.com'],
  development: ['http://localhost:3000', 'http://localhost:5173'],
};

const origins = allowedOrigins[process.env.NODE_ENV] || allowedOrigins.development;

app.use(cors({
  origin: (incomingOrigin, callback) => {
    // Allow server-to-server calls (no origin header)
    if (!incomingOrigin) return callback(null, true);

    if (origins.includes(incomingOrigin)) {
      callback(null, true);
    } else {
      callback(new Error(`CORS: ${incomingOrigin} not allowed`));
    }
  },
  credentials: true,          // allow cookies to be sent
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-CSRF-Token'],
  maxAge: 86400,              // preflight cache: 24 hours
}));
```

## 6. SQL Injection Prevention

### What is it?
Never build SQL queries with string concatenation. Always use parameterized queries.

```js
// ❌ NEVER — attacker enters: ' OR '1'='1
const result = await pool.query(
  `SELECT * FROM users WHERE email = '${email}'`
);

// ✅ ALWAYS — pg sends params separately, SQL structure is fixed
const result = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  [email]
);

// ✅ Multiple params
const result = await pool.query(
  'SELECT * FROM orders WHERE user_id = $1 AND status = $2 AND created_at > $3',
  [userId, status, fromDate]
);

// ✅ With an ORM (Prisma) — parameterized by default
const user = await prisma.user.findUnique({ where: { email } });
```

## 7. Security Headers — Helmet

### What is it?
HTTP headers that tell browsers how to behave safely. One line of code covers 14 headers.

```js
const helmet = require('helmet');

app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", "data:", "https:"],
      connectSrc: ["'self'", "https://api.myapp.com"],
      frameSrc: ["'none'"],
      objectSrc: ["'none'"],
    }
  },
  hsts: {
    maxAge: 31536000,
    includeSubDomains: true,
    preload: true
  },
  frameguard: { action: 'deny' },  // no clickjacking
  noSniff: true,
  hidePoweredBy: true,             // hide "X-Powered-By: Express"
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' }
}));
```

## 9. API Key Security (for External API Access)

### What is it?
When other services or apps call your API (not users), use API keys instead of JWT.

### Implementation

```js
// ── GENERATE API KEY ───────────────────────────────────
const crypto = require('crypto');

function generateApiKey() {
  return 'sk_' + crypto.randomBytes(32).toString('hex');
  // looks like: sk_a3f8c2d1e4b5...
}

// Store HASH in DB — not the raw key
const apiKey = generateApiKey();
const keyHash = crypto.createHash('sha256').update(apiKey).digest('hex');

await pool.query(
  'INSERT INTO api_keys (name, key_hash, user_id) VALUES ($1, $2, $3)',
  ['My App Key', keyHash, userId]
);

// Show raw key to user ONCE — never store it plain
res.json({ apiKey }); // user copies this, you never see it again


// ── VALIDATE API KEY MIDDLEWARE ────────────────────────
async function validateApiKey(req, res, next) {
  const apiKey = req.headers['x-api-key'];

  if (!apiKey) {
    return res.status(401).json({ error: 'API key required' });
  }

  // Hash the incoming key and compare to stored hash
  const keyHash = crypto.createHash('sha256').update(apiKey).digest('hex');

  const result = await pool.query(
    'SELECT * FROM api_keys WHERE key_hash = $1 AND is_active = true AND (expires_at IS NULL OR expires_at > NOW())',
    [keyHash]
  );

  if (!result.rows[0]) {
    return res.status(401).json({ error: 'Invalid or expired API key' });
  }

  // Attach key info to request
  req.apiKey = result.rows[0];

  // Log usage
  await pool.query(
    'UPDATE api_keys SET last_used_at = NOW(), usage_count = usage_count + 1 WHERE id = $1',
    [result.rows[0].id]
  );

  next();
}

// Usage
app.get('/api/v1/data', validateApiKey, dataHandler);
```

## 10. Sensitive Data — Return Only What's Needed

### What is it?
API should never return more data than the client needs. Over-fetching leaks internal data.

```js
// ❌ Returns password hash, admin flags, internal notes
app.get('/api/users/:id', authenticate, async (req, res) => {
  const user = await pool.query('SELECT * FROM users WHERE id = $1', [id]);
  res.json(user.rows[0]); // sends EVERYTHING
});


// ✅ Select only needed fields
app.get('/api/users/:id', authenticate, async (req, res) => {
  const user = await pool.query(
    'SELECT id, name, email, avatar_url, created_at FROM users WHERE id = $1',
    [id]
  );
  res.json(user.rows[0]);
});


// ✅ Different views for different roles
async function formatUserResponse(user, requestingUserRole) {
  const base = {
    id: user.id,
    name: user.name,
    avatar_url: user.avatar_url,
  };

  if (requestingUserRole === 'admin') {
    return { ...base, email: user.email, role: user.role, created_at: user.created_at };
  }

  return base; // public view — minimal data
}
```

## 11. Error Handling — Never Leak Internals

```js
// Custom error class
class AppError extends Error {
  constructor(message, statusCode, isOperational = true) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = isOperational;
  }
}

// Global error handler — last middleware
app.use((err, req, res, next) => {

  // 1. Log full error internally
  console.error({
    error: err.message,
    stack: err.stack,
    path: req.path,
    method: req.method,
    userId: req.user?.userId,
    ip: req.ip,
    timestamp: new Date().toISOString()
  });

  // 2. Operational errors → send specific message
  if (err.isOperational) {
    return res.status(err.statusCode).json({ error: err.message });
  }

  // 3. Unknown errors → hide details in production
  const message = process.env.NODE_ENV === 'production'
    ? 'Something went wrong'
    : err.message;

  res.status(500).json({ error: message });
});

// Throw in handlers like this:
throw new AppError('User not found', 404);
throw new AppError('Email already in use', 409);
throw new AppError('Not authorized', 403);
```

## 12. Request Size Limiting — Stop Large Payload Attacks

```js
// Limit body size (default is 100kb — often way too large)
app.use(express.json({ limit: '10kb' }));
app.use(express.urlencoded({ extended: true, limit: '10kb' }));

// For file uploads — limit separately
const multer = require('multer');
const upload = multer({
  limits: {
    fileSize: 5 * 1024 * 1024,  // 5MB max
    files: 1                     // 1 file at a time
  },
  fileFilter: (req, file, cb) => {
    // Whitelist allowed file types
    const allowed = ['image/jpeg', 'image/png', 'image/webp'];
    if (allowed.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new AppError('Only JPEG, PNG, WebP allowed', 400));
    }
  }
});
```

## 13. Logging & Monitoring — Know When You're Being Attacked

```js
const winston = require('winston');

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
    new winston.transports.File({ filename: 'logs/combined.log' }),
  ]
});

// Log every API request
app.use((req, res, next) => {
  const start = Date.now();

  res.on('finish', () => {
    logger.info({
      method: req.method,
      path: req.path,
      status: res.statusCode,
      duration: Date.now() - start,
      ip: req.ip,
      userId: req.user?.userId,
      // Never log: passwords, tokens, credit cards, SSN
    });

    // Alert on suspicious patterns
    if (res.statusCode === 401 || res.statusCode === 403) {
      logger.warn({
        type: 'AUTH_FAILURE',
        ip: req.ip,
        path: req.path,
        userId: req.user?.userId,
      });
    }
  });

  next();
});

// In production — send logs to Datadog, CloudWatch, Sentry
```

## 14. Secrets Management — Never Hardcode

```js
// ── LOAD AND VALIDATE AT STARTUP ──────────────────────
function requireEnv(key) {
  const value = process.env[key];
  if (!value) throw new Error(`Missing required environment variable: ${key}`);
  return value;
}

const config = {
  jwtSecret:        requireEnv('JWT_SECRET'),
  jwtRefreshSecret: requireEnv('JWT_REFRESH_SECRET'),
  dbUrl:            requireEnv('DATABASE_URL'),
  redisUrl:         requireEnv('REDIS_URL'),
  nodeEnv:          requireEnv('NODE_ENV'),
};
// App crashes at startup if any secret is missing — fail fast

// ── IN PRODUCTION — USE A SECRETS VAULT ───────────────
// AWS Secrets Manager
const { SecretsManagerClient, GetSecretValueCommand } = require('@aws-sdk/client-secrets-manager');

async function getSecret(secretName) {
  const client = new SecretsManagerClient({ region: 'ap-south-1' });
  const response = await client.send(new GetSecretValueCommand({ SecretId: secretName }));
  return JSON.parse(response.SecretString);
}

const dbCreds = await getSecret('prod/myapp/database');
// { host, port, username, password, dbname }
```

## 15. API Versioning — Deprecate Safely

```js
// Version your API from day 1
app.use('/api/v1', v1Router);
app.use('/api/v2', v2Router);

// Deprecation header — warn clients to upgrade
app.use('/api/v1', (req, res, next) => {
  res.setHeader('Deprecation', 'true');
  res.setHeader('Sunset', 'Sat, 01 Jan 2026 00:00:00 GMT');
  res.setHeader('Link', '</api/v2>; rel="successor-version"');
  next();
});
```

Why security? Old API versions with known vulnerabilities stay alive forever without versioning + deprecation strategy. Attackers specifically target old endpoints.

## Everything Together — Secure API Request Flow

```
Client sends: POST /api/orders
              Authorization: Bearer <token>
              Content-Type: application/json
              Body: { "productId": 5, "quantity": 2 }
                     │
                     ▼
         ┌─────────────────────────┐
         │  1. HTTPS               │  Is it encrypted? If not → reject
         │  2. Helmet headers      │  Security headers injected
         │  3. CORS check          │  Is origin allowed?
         │  4. Rate limiter        │  Too many requests? → 429
         │  5. Body size check     │  Over 10kb? → 413
         │  6. authenticate()      │  Valid JWT? → decode user
         │  7. authorize()         │  Has 'buyer' role?
         │  8. Input validation    │  productId = integer? quantity = 1–10?
         │  9. Ownership check     │  Does product exist and is active?
         │ 10. Business logic      │  Enough stock?
         │ 11. Parameterized SQL   │  INSERT INTO orders...
         │ 12. Select only needed  │  Return { orderId, total, status }
         │ 13. Log the request     │  method, path, userId, duration
         └─────────────────────────┘
                     │
                     ▼
              ✅ 201 Created
              { "orderId": 42, "total": 99.98, "status": "pending" }
```

## API Security Checklist

```
Authentication
  ✅ JWT verified on every protected request
  ✅ Short-lived access tokens (15 min)
  ✅ Refresh tokens stored in HttpOnly cookie
  ✅ Refresh tokens stored hashed in DB (revocable)
  ✅ Logout deletes refresh token from DB

Authorization
  ✅ Role checked before every sensitive action
  ✅ Ownership checked before returning any resource
  ✅ Field-level access control on updates

Input
  ✅ All body/params/query validated and sanitized
  ✅ All SQL queries parameterized
  ✅ Request body size limited (10kb)
  ✅ File type and size validated on uploads

Transport
  ✅ HTTPS only — HTTP redirected
  ✅ TLS 1.2+ only
  ✅ HSTS header with preload
```

## Interview Answer

> *"API security is about making sure every request is verified and every piece of data is protected, regardless of how the request comes in — whether from my React app or directly from Postman.*
>
> *I start with authentication — every protected route goes through a JWT middleware that verifies the token signature and expiry. Then authorization — I check role permissions AND resource ownership to prevent IDOR attacks.*
>
> *I validate all incoming data server-side with express-validator, use parameterized queries for every DB call to prevent SQL injection, and apply Helmet for security headers. Rate limiting sits on all routes — stricter on auth endpoints.*
>
> *For transport, everything is HTTPS with HSTS. CORS is a strict origin whitelist. Secrets live in environment variables, never in code. API responses only return fields the client actually needs — never SELECT *.*
>
> *The mindset is: the frontend is untrusted. Every real security check lives in the API."*
