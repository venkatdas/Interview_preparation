# WCAG (Web Content Accessibility Guidelines) - Complete Guide

## üéØ What is WCAG? (In Simple Words)

**WCAG** = Rules to make websites usable by **everyone**, including people with disabilities.

### Real-World Analogy
Just like buildings have **ramps for wheelchairs**, websites need **accessibility features** for people who:
- Can't see (blind)
- Can't hear (deaf)
- Can't use a mouse (motor disabilities)
- Have difficulty reading (dyslexia, cognitive disabilities)

## üéØ The 4 Main Principles (POUR)

Remember: **POUR** (like pouring water - flows to everyone!)

"I can **see** or **hear** the content"

"I can **use** and **navigate** it"

## 1Ô∏è‚É£ PERCEIVABLE

### 1.1 Text Alternatives

**Rule:** Provide text alternatives for non-text content

#### ‚ùå Bad Example:
```html
<!-- Screen reader reads: "image" - useless! -->
<img src="cat.jpg">
```

#### ‚úÖ Good Example:
```html
<!-- Screen reader reads: "A fluffy orange cat sleeping on a couch" -->
<img src="cat.jpg" alt="A fluffy orange cat sleeping on a couch">
```

#### More Examples:
```html
<!-- Decorative images (don't add meaning) -->
<img src="decorative-line.png" alt="" role="presentation">

<!-- Icon buttons -->
<button>
  <img src="save-icon.svg" alt="Save document">
</button>

<!-- Complex images (charts, diagrams) -->
<img src="sales-chart.png" alt="Sales chart showing 50% increase in Q4">
<!-- Also provide long description -->
<p>Detailed description: Sales increased from $100k in Q1 to $150k in Q4...</p>

<!-- Background images with important content -->
<div role="img" aria-label="Welcome banner with company logo">
  <!-- Background image via CSS -->
</div>
```

### 1.2 Time-Based Media (Videos & Audio)

**Rule:** Provide captions, transcripts, and audio descriptions

#### ‚úÖ Good Example:
```html

<video controls>
  <source src="tutorial.mp4" type="video/mp4">
  <track kind="captions" src="captions-en.vtt" srclang="en" label="English">
  <track kind="descriptions" src="descriptions.vtt" srclang="en" label="Audio descriptions">

  <summary>Video Transcript</summary>

</details>
```

**Real-world:** YouTube auto-captions, Netflix subtitles

### 1.3 Adaptable Content

**Rule:** Content can be presented in different ways without losing information

#### ‚ùå Bad Example:
```html
<!-- Using visual formatting only -->
<div style="color: red; font-weight: bold;">Error!</div>
<p>Please fix the following:</p>
```

#### ‚úÖ Good Example:
```html
<!-- Semantic HTML + proper structure -->
<div role="alert" aria-live="assertive">
  <strong>Error!</strong>
  <p>Please fix the following:</p>
  <ul>
    <li>Email is required</li>
    <li>Password must be 8+ characters</li>
  </ul>
</div>
```

#### Proper Heading Structure:
```html
<!-- ‚ùå BAD: Skipping heading levels -->
<h1>Main Title</h1>
<h3>Subsection</h3> <!-- Skipped h2! -->

<!-- ‚úÖ GOOD: Logical hierarchy -->
<h1>Main Title</h1>
<h2>Section</h2>
<h3>Subsection</h3>
<h4>Sub-subsection</h4>
```

### 1.4 Distinguishable (Colors & Contrast)

**Rule:** Make it easy to see and hear content

#### Color Contrast Ratios:
```
Normal text (under 18pt): 4.5:1 (AA) or 7:1 (AAA)
Large text (18pt+ or 14pt+ bold): 3:1 (AA) or 4.5:1 (AAA)
```

#### ‚ùå Bad Example:
```html
<!-- Light gray text on white background - hard to read -->
<p style="color: #ccc; background: #fff;">
  Important information
</p>
<!-- Contrast ratio: 1.6:1 - FAILS! -->
```

#### ‚úÖ Good Example:
```html
<!-- Dark text on light background - easy to read -->
<p style="color: #333; background: #fff;">
  Important information
</p>
<!-- Contrast ratio: 12.6:1 - PASSES! -->
```

#### Don't Rely on Color Alone:
```html
<!-- ‚ùå BAD: Color only -->
<p style="color: red;">Required field</p>

<!-- ‚úÖ GOOD: Color + icon + text -->
<p style="color: red;">
  <span aria-label="required">*</span>
  <strong>Required</strong> field
</p>
```

#### Tools to Check Contrast:
```javascript
// Online tools:
// 1. WebAIM Contrast Checker
// 2. Contrast Ratio by Lea Verou
// 3. Chrome DevTools (built-in)

// Example contrast calculation:
const foreground = '#333333'; // Dark gray
const background = '#FFFFFF'; // White
// Contrast ratio: 12.63:1 ‚úÖ PASSES AA & AAA
```

## 2Ô∏è‚É£ OPERABLE

### 2.1 Keyboard Accessible

**Rule:** All functionality must be available via keyboard

#### ‚ùå Bad Example:
```html
<!-- Only works with mouse! -->
<div onclick="doSomething()">Click me</div>
```

#### ‚úÖ Good Example:
```html
<!-- Works with mouse AND keyboard -->
<button onclick="doSomething()">Click me</button>

<!-- Or custom element with keyboard support -->
<div 
  role="button" 
  tabindex="0"
  onclick="doSomething()"
  onkeypress="handleKeyPress(event)">
  Click me
</div>

<script>
function handleKeyPress(event) {
  // Trigger on Enter or Space
  if (event.key === 'Enter' || event.key === ' ') {
    event.preventDefault();
    doSomething();
  }
}
</script>
```

#### Keyboard Navigation Test:
```
Tab ‚Üí Move forward
Shift+Tab ‚Üí Move backward
Enter ‚Üí Activate button/link
Space ‚Üí Activate button, toggle checkbox
Arrow keys ‚Üí Navigate menus, radio buttons
Esc ‚Üí Close dialog/menu
```

### 2.2 Enough Time

**Rule:** Users have enough time to read and use content

#### ‚ùå Bad Example:
```javascript
// Auto-logout after 60 seconds with no warning
setTimeout(() => {
  logout();
}, 60000);
```

#### ‚úÖ Good Example:
```javascript
// Show warning 20 seconds before timeout
let timeoutId;
let warningId;

function startTimer() {
  // Warn after 10 minutes
  warningId = setTimeout(showWarning, 10 * 60 * 1000);
  // Logout after 12 minutes
  timeoutId = setTimeout(logout, 12 * 60 * 1000);
}

function showWarning() {
  const modal = document.createElement('div');
  modal.innerHTML = `
    <div role="alertdialog" aria-labelledby="timeout-title">
      <h2 id="timeout-title">Session Expiring Soon</h2>
      <p>Your session will expire in 2 minutes. Do you want to continue?</p>
      <button onclick="extendSession()">Continue Session</button>
      <button onclick="logout()">Logout</button>
    </div>
  `;
  document.body.appendChild(modal);
}

function extendSession() {
  clearTimeout(timeoutId);
  clearTimeout(warningId);
  startTimer(); // Restart timer
}
```

### 2.3 Seizures and Physical Reactions

**Rule:** Don't use flashing content that could cause seizures

#### ‚ùå Bad Example:
```javascript
// Flashing animation - DANGEROUS!
setInterval(() => {
  element.style.backgroundColor = 
    element.style.backgroundColor === 'red' ? 'blue' : 'red';
}, 100); // Flashing 10 times per second - FAILS!
```

#### ‚úÖ Good Example:
```javascript
// Smooth transitions
element.style.transition = 'background-color 2s ease';
element.style.backgroundColor = 'blue';

// If flashing is necessary, keep it under 3 flashes per second
// and make it small (<25% of screen)
```

**Rule:** No more than **3 flashes per second**

### 3.3 Input Assistance

**Rule:** Help users avoid and correct mistakes

#### ‚úÖ Error Identification:
```html
<form>
  <label for="email">Email Address *</label>
  <input 
    type="email" 
    id="email" 
    required 
    aria-required="true"
    aria-invalid="false"
    aria-describedby="email-error">
  
  <!-- Error message (hidden until needed) -->
  <div id="email-error" class="error" hidden>
    <strong>Error:</strong> Please enter a valid email address.
  </div>
</form>

<script>
function validateEmail() {
  const input = document.getElementById('email');
  const error = document.getElementById('email-error');
  
  if (!input.value.includes('@')) {
    input.setAttribute('aria-invalid', 'true');
    error.hidden = false;
  } else {
    input.setAttribute('aria-invalid', 'false');
    error.hidden = true;
  }
}
</script>
```

#### ‚úÖ Error Prevention:
```html
<!-- Confirmation before destructive action -->
<button onclick="confirmDelete()">Delete Account</button>

<script>
function confirmDelete() {
  const confirmed = confirm(
    'Are you sure you want to delete your account? This cannot be undone.'
  );
  
  if (confirmed) {
    deleteAccount();
  }
}
</script>
```

#### ‚úÖ Helpful Labels and Instructions:
```html
<label for="password">
  Password
  <span class="hint">(Must be at least 8 characters)</span>
</label>
<input 
  type="password" 
  id="password"
  aria-describedby="password-requirements">

<ul id="password-requirements">
  <li>At least 8 characters</li>
  <li>Include uppercase and lowercase</li>
  <li>Include at least one number</li>
</ul>
```

## üéì Interview Questions & Answers

### Q1: What is WCAG?

**Answer:**
"WCAG stands for Web Content Accessibility Guidelines. It's a set of international standards created by W3C to make websites accessible to everyone, including people with disabilities. The current version is WCAG 2.1, with 2.2 being the latest.

WCAG is built on four principles called POUR:
- Perceivable: Content must be presentable to users
- Operable: Users must be able to operate the interface
- Understandable: Information must be understandable
- Robust: Content must work with assistive technologies

The goal is Level AA compliance, which is the industry standard and legal requirement in many countries."

---

### Q2: What are the three conformance levels?

**Answer:**
"WCAG has three levels:

**Level A (Basic)** - Minimum accessibility, addresses major barriers. About 30 criteria.

**Level AA (Recommended)** - This is the industry standard and what most companies aim for. It includes all Level A criteria plus 20 more. This is legally required in most countries.

**Level AAA (Enhanced)** - The highest level, very difficult to achieve for all content. Includes all AA criteria plus 28 more.

In practice, we typically target Level AA because it balances thoroughness with practicality, and it's what laws like ADA require."

---

### Q3: How do you test for accessibility?

**Answer:**
"I use a combination of automated and manual testing:

**Automated tools:**
- Lighthouse in Chrome DevTools for quick audits
- axe DevTools extension for detailed WCAG violations
- WAVE for visual representation of issues
- Pa11y for continuous integration testing

**Manual testing:**
- Keyboard-only navigation (unplug mouse, try using Tab, Enter, arrows)
- Screen reader testing with NVDA (Windows) or VoiceOver (Mac)
- Color contrast checking with WebAIM Contrast Checker
- Zoom to 200% to ensure text reflows properly
- Test with browser extensions that simulate color blindness

Automated tools catch about 30-40% of issues, so manual testing is crucial."

---

### Q4: Give examples of common accessibility issues.

**Answer:**
"Common issues I've seen and fixed:

1. **Missing alt text** - Images without alternative text for screen readers
   Fix: Add descriptive alt attributes

2. **Low contrast** - Light gray text on white background
   Fix: Ensure 4.5:1 ratio for normal text

3. **Missing form labels** - Inputs without associated labels
   Fix: Use proper `<label>` elements with `for` attribute

4. **Keyboard inaccessible** - Divs with click handlers
   Fix: Use semantic `<button>` elements or add proper keyboard handling

5. **Auto-playing videos** - Videos that start automatically
   Fix: Add controls and don't autoplay

6. **Poor heading structure** - Skipping from h1 to h3
   Fix: Use logical heading hierarchy (h1 ‚Üí h2 ‚Üí h3)"

---

### Q5: What is ARIA and when should you use it?

**Answer:**
"ARIA stands for Accessible Rich Internet Applications. It's a set of attributes you add to HTML to make dynamic content and custom components accessible.

**When to use ARIA:**
- Custom widgets that don't have semantic HTML equivalents (like complex dropdowns, tabs, carousels)
- Dynamic content updates (aria-live regions)
- To provide additional context (aria-label, aria-describedby)

**Important rules:**
1. Use semantic HTML first - don't use ARIA if HTML can do it
2. Don't change native semantics (don't put role=button on an `<h1>`)
3. All interactive elements must be keyboard accessible
4. Roles must be used correctly

**Example:**
```html
<!-- Good: Custom accordion -->
<button 
  aria-expanded="false" 
  aria-controls="panel1">
  Section 1
</button>
<div id="panel1" hidden>
  Content
</div>
```

The first rule of ARIA is: Don't use ARIA if you don't have to!"

---

### Q6: What's the difference between semantic HTML and ARIA?

**Answer:**
"Semantic HTML uses built-in HTML elements that have inherent meaning and accessibility features. ARIA adds accessibility information to elements that don't have it naturally.

**Semantic HTML (always prefer this):**
```html
<button>Click me</button>
<nav>...</nav>
<main>...</main>
```

**ARIA (use when semantic HTML isn't enough):**
```html
<div role="button" tabindex="0">Click me</div>
<div role="navigation">...</div>
```

Semantic HTML is better because:
- It works automatically with assistive technologies
- Provides keyboard interaction for free
- More reliable and less error-prone
- Better browser support

Only use ARIA when you're building custom components that can't be achieved with standard HTML."
